Libraries

Methods

Our Graphics library inherits from the IDisplayModule interface and is composed of a series of precise methods

  • void initWindow() : Initialize the game window

  • bool isOpen() : return True if the window is opened, else return false if the window is closed or not initialized

  • char inputListener() : Listening to all keyboard and return a key index when a pressed key is detected. See all references here.

  • void eventListener() : Listening to any event that could affect the game window (eg: close the window)

  • sf::Sprite *createEntity(int x, int y, char *path) : TODO invalid method

  • void oneCycleDisplay() : display elements during one cycle

Entity recognition

The Graphical library only accepts certain types of GameObject. It will react according to the type of each.

  • WALL : Interpreted as an obstacle

  • FLOOR : Interpreted as path

  • PLAYER : Interpreted as the main player, following the path and reacting to obstacles and enemies

  • ENEMY : Interpreted as the an enemy, following the path and reacting to obstacles and main player

  • BONUS: Interpreted as a collectible

For more information about the TyoeEntity, follow this link.

Dynamic libraries

You must use dynamic libraries at run-time. This means that you MUST use the dlopen, dlclose, dlsym and dlerror functions to handle your dynamic libraries.

Running ldd on your program must not show any dependency towards your libraries
The dynamic libraries MUST be named according to the graphics or game library they implement.

These dynamic libraries can be seen as plug-ins providing various capabilities to your main program. In NO CASE must graphic libraries influence the game logic. Game libraries must not contain any information about screen rendering or low-level events.

You MUST NOT differentiate your libraries. Each of your libraries MUST be handled in a generic and uniform manner. Keep genericity!
